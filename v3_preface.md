# ssiLang v3 Preface

ssiLang 到 `v2.2` 为止，已经从“单文件玩具解释器”跨过了一个关键门槛：

- 有基本完整的表达式和控制流
- 有闭包、容器、引用和安全访问
- 有 `%net` / `&tcp` / `&pipe` / `&shm`
- 有多文件开发能力
- 有能支撑继续演化的测试和教程

但 `v2.2` 仍然非常明显地停留在“可用的实验语言”阶段，而不是“可以持续扩展的正式语言实现”阶段。

`v3` 的任务不是继续零散加语法糖，而是把 ssiLang 从：

```text
树解释器 + 动态值系统 + 小型实验语言
```

推进到：

```text
有执行模型、有模块边界、有性能路线、有工具链意识的小型正式语言
```

这份文档不是发布说明，而是下一版的总前言和设计路线。

---

## 1. v3 的核心目标

v3 不是追求“大而全”，而是追求三件事：

1. 执行模型清晰
2. 工程组织清晰
3. 对外协作清晰

换句话说，v3 要解决的不是“再多几个功能”，而是下面这些根问题：

- 现在的执行器是 AST 直解释，性能和可优化性都有限
- 模块能力刚刚起步，还缺少正式导出机制和项目结构约定
- IPC 和 `%net` 已经可用，但跨语言协作仍缺少官方参考实现
- 错误信息、调试能力、工具链还不够工程化

---

## 2. v3 的设计原则

### 2.1 保持语言小

ssiLang 不应该走向“大而杂”的脚本语言。v3 依然要刻意保持小：

- 小关键字集合
- 小语法核心
- 少魔法
- 少隐式规则

新增能力优先放在：

- 更清晰的执行模型
- 更可靠的模块和工具
- 更稳的对外协议

而不是无止境堆表面语法。

### 2.2 偏运行时一致性，而不是语法花样

如果一个功能能通过更清晰的 runtime 语义解决，就不要优先发明新语法。

例如：

- 模块系统优先补 `export` / `from import`
- 性能优先补字节码和槽位
- 协作优先补 `%net` V1 参考实现

而不是先去发明复杂宏系统、装饰器系统、模式匹配语法等。

### 2.3 明确支持“数据型动态语言”

ssiLang 的强项不是元编程，也不是复杂 OO。

它的最合理定位是：

- 数据变换
- 算法脚本
- 进程协作
- 小型服务逻辑
- 跨进程状态和消息编排

所以 v3 应该围绕“数据 + 函数 + 协作”演化，而不是硬拗成另一种 Python/JS。

---

## 3. v3 的版本主题

如果用一句话概括：

> v3 = 执行模型升级 + 模块系统成型 + 协作协议正式化

具体拆成三条主线：

- Runtime line: 从树解释走向字节码 VM
- Language line: 从 `import()` 走向正式模块系统
- Tooling line: 从“能跑”走向“能开发、能调试、能协作”

---

## 4. Runtime 路线

### 4.1 从 AST 解释器到 Bytecode VM

这是 v3 最重要的内部工程。

当前实现是：

```text
源码 -> Lexer -> Parser -> AST -> 递归解释执行
```

v3 目标是：

```text
源码 -> Lexer -> Parser -> AST -> Bytecode -> VM
```

理由很直接：

- AST 解释开销高
- 很难做热点优化
- 函数调用和局部变量访问成本偏高
- 后续若要做 JIT，字节码层是更合理的中间层

v3 不要求一步到位上 JIT，但至少要把执行模型升级到可优化状态。

### 4.2 局部变量槽位化

当前环境模型大量依赖字符串查找。

v3 应该把：

- 局部变量
- 函数参数
- 闭包捕获

尽量编译成槽位访问。

这会立刻带来三类收益：

- 更低的运行时查找成本
- 更稳定的调用开销
- 为闭包、调试信息、优化 passes 提供更清晰的 IR 基础

### 4.3 调用点缓存

v3 可以引入轻量级 inline cache：

- map 成员访问缓存
- 方法调用缓存
- 全局函数查找缓存

这不是完整 JIT，但对动态语言非常值。

### 4.4 Value 系统重整

当前 `Value` 是合理的起点，但 v3 应该开始明确：

- 哪些值是 plain data
- 哪些值是 runtime handle
- 哪些值允许序列化
- 哪些值允许共享

长远看，v3 最好把“可序列化值集”和“运行时句柄值集”在文档和实现上都进一步收紧。

---

## 5. 模块与项目系统

### 5.1 从 `import(path)` 走向正式导出

`v2.2` 的 `import(path)` 已经够用，但还不够正式。

v3 计划增加：

- `export name = ...`
- `export fn f(...) { ... }`
- `from "x" import foo, bar`

目标不是搞复杂模块语法，而是明确边界：

- 哪些名字公开
- 哪些名字私有
- 哪些导入是具名依赖

### 5.2 项目入口约定

v3 应该给项目结构一个官方约定，例如：

```text
project/
  ssi.toml
  src/
    main.sl
    lib/
  test/
```

不一定要立刻做完整包管理器，但至少要有：

- 入口文件约定
- 源码目录约定
- 测试目录约定
- 模块搜索路径约定

### 5.3 模块缓存与循环导入语义正式化

这部分在 `v2.2` 已经有实现，但 v3 要把语义写死：

- 模块加载时机
- 缓存命中规则
- 循环导入报错规则
- 模块单例语义

这会让模块系统从“实现行为”变成“语言规范”。

---

## 6. 协作与协议路线

### 6.1 `%net V1` 正式规范化

`v2.2` 已经把 `%net` 讲清楚了，但 v3 应该进一步做成真正的规范资产：

- 单独文档
- 版本字段约束
- little-endian framing 约束
- 错误输入处理规则
- 兼容性说明

目标是让外部语言实现者可以独立写解码器，而不必反读解释器源码。

### 6.2 官方跨语言参考实现

v3 建议至少提供：

- Python V1 codec
- C++ V1 codec
- Rust V1 codec

这样 `%net` 才会从“ssiLang 内部格式”变成“可以实际协作的协议”。

### 6.3 IPC 选型指南升级

v3 应该把 IPC 文档进一步工程化：

- `&tcp` 适合什么
- `&pipe` 适合什么
- `&shm` 适合什么
- 什么场景不要滥用 JSON
- 什么场景不要滥用共享内存

最好附上：

- 延迟/吞吐直觉
- 状态同步 vs 消息通信对比
- 常见错误案例

---

## 7. 语言层面的计划特性

以下是 v3 可以合理纳入的语言改进。

### 7.1 正式 `export`

这是最优先的语言特性之一。

### 7.2 多行 lambda 暂缓

v3 不建议急着加多行 lambda。

原因：

- 命名函数已经够用
- 多行 lambda 会引入新的语法和作用域复杂度
- 对当前语言收益不如模块和运行时升级大

### 7.3 模式匹配暂缓

这是很诱人的功能，但不应该抢占 v3 的主线。

### 7.4 更好的错误定位

v3 应该增强：

- 错误文件名
- 行号
- 上下文代码片段
- import 链路
- 调用栈

如果没有这些，再多语法糖也只是让问题更难查。

### 7.5 REPL 改进

v3 可考虑：

- 多行输入
- 历史记录
- 更清晰的错误提示
- 当前模块上下文显示

---

## 8. 工具链计划

### 8.1 `sl test`

v3 应该有一个内建测试入口，而不是完全依赖外部脚本。

例如：

```text
sl test
sl test test/t31_zero_arity_lambda.sl
```

### 8.2 `sl fmt`

如果语言准备继续发展，格式化器迟早要有。

尤其是：

- lambda
- split 表达式
- 长管道
- map / list 多行写法

没有 formatter，代码风格会很快漂移。

### 8.3 `sl check`

即使动态语言，也可以先做轻量静态检查：

- 重复定义提醒
- 未使用变量提醒
- 明显死代码
- 无法序列化的 `%net` 用法
- `send` / `store` 未传 net-form

这类检查对 LLM 和人类都很有价值。

### 8.4 调试与反汇编

如果引入 bytecode，v3 最好顺手提供：

- `sl dump-ast`
- `sl dump-bc`
- `sl trace`

这对语言开发本身会极其有帮助。

---

## 9. 性能目标

v3 不需要和成熟 JIT 语言竞争，但至少应该达到“明显好于当前树解释器”的水平。

建议定三个现实目标：

1. 算法脚本性能显著提升
2. 高频函数调用开销显著下降
3. map/member 访问成本明显下降

如果做 benchmark，建议覆盖：

- 递归 fib
- 闭包计数器
- list map/filter/reduce
- 大量 map 访问
- `%net` 编解码
- tcp/pipe/shm 往返

---

## 10. 兼容性策略

v3 应坚持：

- 默认向后兼容 `v2.2` 脚本
- 新语法优先增量引入
- `%net V1` 在 v3 中保持可读可写

如果未来要出 `%net V2`，策略应该是：

- v3 默认仍支持 V1
- 文档明确版本协商方式
- 工具链提供格式转换或探测能力

---

## 11. 不在 v3 的内容

为了避免失控，以下内容明确不作为 v3 主线：

- 完整 JIT 编译器
- 泛型类型系统
- 复杂 class / inheritance 系统
- 宏系统
- 模板元编程
- 多线程并发运行时
- 大型标准库膨胀

这些不是永远不做，而是现在不该抢占资源。

---

## 12. 建议的里程碑

### v3.0-alpha

- Bytecode 设计定稿
- AST -> bytecode 编译跑通
- VM 可执行基础脚本
- `dump-bc` 可用

### v3.0-beta

- 模块系统补 `export`
- `%net` V1 独立规范文档
- Python 参考实现
- 更完整错误信息

### v3.0-rc

- 全量回归迁移
- 文档重写完成
- benchmark 建立
- `sl test` / `sl check` 初版

### v3.0

- 默认执行器切到 VM
- `v2.2` 核心脚本兼容通过
- 对外协议和项目结构稳定

---

## 13. 对实现者的要求

v3 的关键不是“继续写功能”，而是技术纪律：

- 每加一项能力，都要同步教程和测试
- 每个 IPC/协议结论，都要能在源码和测试中自证
- 不能让解释器实现细节和教程长期分叉
- 新性能优化要有 benchmark，不接受纯感觉优化

ssiLang 如果要继续成长，v3 必须是“收敛的一版”，不是“发散的一版”。

---

## 14. 结语

`v2.2` 解决的是“这门语言能不能像样地写东西”。

`v3` 要解决的是另一个更关键的问题：

> 这门语言能不能以一种可持续、可维护、可协作的方式继续进化。

如果 v3 做成，那么 ssiLang 就不再只是一个有趣的解释器项目，而会开始具备真正语言工程的骨架。
